## 2. コンセプト：何の「お悩み」を解決するのか

### ターゲット
Substackで日本語の発信をしている個人（特にこれから伸ばしたい人）。

### 解決する悩み
> 「Substackで毎日書いているけど、**自分の発信は他と比べてどうなのか／誰が伸びていて、何が
> 上位に入る条件なのか** が分からない」

（追記）
Substackで登録・フォローをして読みたいけど、誰が良い記事を出しているかを知りたい

Substack番付（substackbanzuke.com）は日々のTOP30を出してくれるが、
- **その日のスナップショット**しか見えず、「**誰が継続的に強いのか**」が分からない
- 発行元（書き手）単位での通算成績が一覧できない
- 過去にさかのぼって傾向を追えない

### 提供価値（このサイトがやること）
日々のTOP30を **累積** して、**発行元（書き手）別のハイランカー表** を見せる。
- 「**ベンチマーク**」… 自分や憧れの書き手が、通算でどのくらい上位常連なのか一目で分かる
- 「**ロールモデル発見**」… 継続的に強い書き手＝学ぶべき人をランキングから見つけられる
- 「**戦略のヒント**」… どのカテゴリ・どんな書き手が上位に多いかが分かる
↑
（追記）
月毎の個人個人のランキング
5月はこうだった
6月はこうだった
今月はこうだった
直近30日はこうだった
直近7日はこうだった
トータル累積はこうだった
あたりが見れるようになったらいいなと思っています。

> キャッチコピー案（要検討）：「番付の"その日"を、"通算"で見る。Substackで伸びている人が分かるサイト。」

## 3. データ

### 3.1 ソース
- 取得元: `https://substackbanzuke.com/daily/{date}`（日次TOP30）
- 既存スクレイパー: `ruku_data/scripts/banzuke_scraper.py`
- 既存データ: `ruku_data/Substack_posts/analytics/banzuke_full.csv`
  - 期間: **2026-05-07 〜 現在**（毎日蓄積中、約1,400行）
  - 発行元（ユニーク）: **約242件**

### 3.2 日次データの列（banzuke_full.csv）
```
fetched_at, date, rank, title, publisher, url,
attention_score, likes, restacks, comments, category, is_ruku
```

### 3.3 週次データ（任意で活用）
- `banzuke_weekly_full.csv`（週次TOP30）も存在。第2タブ等で使える。

### 3.4 発行元別ランキングの集計ロジック（添付画像の表）
日次データを発行元ごとに集計：

| 表示名 | 定義 |
|---|---|
| 登場日数 | その発行元がTOP30に入った延べ日数 |
| 1位 | rank==1 の回数 |
| Top3 | rank<=3 の回数 |
| Top10 | rank<=10 の回数 |
| 平均順位 | 登場時のrankの平均 |

- 既定の並び順: **Top10入り回数の降順**（＝画像と同じ）
- 既存実装が `scripts/substack_dashboard.py` 内にあり（発行元別Top30）、ロジックを流用可能。

- **A5. 日次TOP30ビューア**（特定日のランキングを見る）

> ⚠️ A1〜A6は審査の「工夫・こだわり」「実用性」に効く。MVP後に優先度を決める。

- ユーザー登録・ログイン
- 番付データの編集・投稿機能
- ルク個人のSubstack分析ダッシュボード（既存のローカルダッシュボードとは別物）

### 4.4 追加であると良さそうな機能（A1〜A6後）
- **B1. 急上昇ランキング**: 直近7日と直近30日/累積を比べて、最近伸びている発行元を発見できるようにする。
- **B2. 発行元の比較**: 2人の順位推移・Top10率・平均注目度を並べて見られるようにする。
- **B3. SNS共有用OGP画像**: X等で共有したときにサイト名・集計期間・特徴が伝わる画像を出す。
- **B4. ランキングCSVダウンロード**: 発行者自身が分析・記録に使えるように、現在表示中のランキングをCSVで保存できるようにする。
- **B5. カテゴリ深掘り**: カテゴリをクリックすると、そのカテゴリ内の強い発行元・日別記事を見られるようにする。
- **B6. はじめて番付入りした発行元**: 新規参入者を見つけやすくし、読む人の発見性を高める。
- **B7. スコア説明の透明化**: 「番付偏差値」が何を元に出ているかを短く説明し、恣意性を減らす。
- **B8. 発行元比較ビュー**: 2つの発行元を選んで、順位推移・Top10率・平均順位・偏差値を並べて比較できるようにする。
- **B9. カテゴリ深掘りパネル**: カテゴリを選ぶと、そのカテゴリで強い発行元と最新の番付入り記事を追えるようにする。
- **B10. 初登場発行元ハイライト**: この期間で初めて番付入りした発行元をまとめて見せ、変化の発見性を上げる。

## 5. 「毎朝自動収集」の方式（最重要の設計判断）

サイト自体が毎朝自動で最新データを取り込む必要がある。候補は3つ：

### 案A: GitHub Actions（cron）＋ 静的サイト 〔推奨〕
- GitHubの定期実行（毎朝）でスクレイパーを走らせ、`data.json` を生成→リポジトリにコミット
- 静的サイト（GitHub Pages / Cloudflare Pages）がそのJSONを読んで表示
- **長所**: 完全無料・サーバー不要・PCを起動していなくても動く・公開維持が容易（審査要件◎）
- **短所**: GitHubアカウントとリポジトリが要る

### 案B: ローカルcron（現状の延長）＋ 生成物をどこかに公開
- 既存の08:05 cronで収集→HTMLを生成→Netlify等へ自動アップロード
- **長所**: 既存の仕組みを最大流用
- **短所**: **Macが起動していないと更新されない**。コンテストの「公開維持」リスクあり

### 案C: サーバー上で動的に収集・配信（VPS/Cloud Run等）
- **長所**: 柔軟　**短所**: コスト・運用が重い。今回はオーバースペック

→ **推奨は案A**（公開維持・無料・自動の三拍子）。最終決定は質問Q2で。

## 6. 技術スタック（案A前提のたたき台）

- **データ収集**: 既存 `banzuke_scraper.py` を流用（JSON出力を追加）
- **集計**: Python（発行元別集計を `build_site_data.py` として切り出し、`data.json` を生成）
- **フロント**: 静的HTML + CSS + 素のJavaScript（フレームワーク無しでも実現可能）
  - もしくは軽量に：Vanilla JS でJSONを fetch して表描画＋ソート＋検索
- **ホスティング**: GitHub Pages もしくは Cloudflare Pages（独自ドメインも可）
- **自動化**: GitHub Actions（`schedule:` cron、毎朝JST）

> 技術力は評価対象外なので、**シンプルな静的サイト + JSON** で十分。凝るなら見た目とUXに投資する。

## 10. 仕様確定サマリー（いただいた回答の反映）

### 10.1 コンセプト（確定）
2種類の「お悩み」を解決するサイトにする：
- **書き手の悩み**: 「自分の発信は他と比べてどうか／誰が継続的に伸びているか」
  → 通算・期間別の発行元ランキングでベンチマークできる
- **読者の悩み**: 「Substackで良い記事を出している人を見つけてフォローしたい」
  → ランキング＋各人のSubstackへのリンクで“良い書き手”を発見できる
- さらに **番付トレンド**（今どんな記事・ジャンルが伸びているか、上位の平均注目度など）も見せる

> サイトの軸 = 「番付の“その日”を、“通算・期間・トレンド”で見える化する非公式スタッツサイト」

### 10.2 機能（確定スコープ）

**MVP（最初に出す。明日〜明後日 6/24〜25 目標）**
- 発行元別ランキング表（**全件表示**・添付画像準拠の列）
  - 列: 発行元 / 登場日数 / 1位 / Top3 / Top10 / 平均順位（＋最高順位）
  - 既定ソート: Top10回数の降順（同点は平均順位の昇順 → 登場日数の降順）
  - **列ヘッダクリックで並べ替え**
  - 発行元名は**各人のSubstackトップへのリンク**にする
  - **ルクの特別ハイライトは廃止**（全発行元フラットに表示）
- 集計期間の表示
- お悩み解決メッセージ（書き手向け＋読者向けの2行）
- スマホ最適化（添付はスマホ表示）

**期間切り替え（Q1追記の要望・MVP直後に実装する中核機能）**
同じ発行元別ランキングを、以下の期間で切り替えて見られる：
- 累積（全期間） / 今月 / 直近30日 / 直近7日
- 月別（2026-05, 2026-06 … データに応じて自動で増える）

**番付トレンド（Q1追記）**
- 順位帯別の平均指標（1位/2位/3位… の平均注目度・♥・Restack）
- カテゴリ別ランキング（どのジャンルが上位に多いか／直近で伸びているか）
- 「直近で伸びている発行元」ピックアップ（直近7日で上位常連 等）

**段階実装の順番**（Q6: まずMVP→1個ずつ）
1. MVP（累積の発行元別ランキング＋リンク＋スマホ対応） … 最優先で公開
2. 期間切り替え（今月/直近30日/直近7日/月別）
3. 番付トレンド（平均注目度・カテゴリ）
4. 発行元クリックで詳細（順位推移グラフ・登場履歴）
5. 検索／絞り込み

### 10.3 データ集計の確定ルール
- **名寄せ（重要）**: 発行元は「`｜` または ` | `（全角/半角パイプ）の前のベース名」でグルーピングする。
  例: 「ちなみ｜書を綴る道。」と「ちなみ｜書を綴る夜。」は**同一人物**として合算。
  - 表示名は **その期間で最新日付の表記**（例: 最新の「ちなみ｜書を綴る夜。」）をフル表示。
- **リンク**: 各発行元の最新記事URLの**ドメイン（パブリケーションのトップ）**へリンク。
  例: `https://rukupractice.substack.com/p/20` → `https://rukupractice.substack.com/`
- **記事単位は出さない**（Q5: 番付本体を見ればよい）。あくまで集計サイト。
- 出典として substackbanzuke.com を明記し、**非公式**である旨を記載。

（追記）上記でOKです。途中で名前やサイト名を変える人は出てくると思うので、毎朝確認し、変わった場合は、URLで名寄せして、全体を変更してもらえればと思います。

### 10.4 自動収集アーキテクチャ（確定 = 案A）
> **Q2の回答: GitHubでできます。** PCが起動していなくても、GitHubのサーバー上で毎朝自動実行されます。
